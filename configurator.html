Deployment Mode

How many AWS accounts will be used for migration?

Single Account

Migration workloads in one AWS account. Deploy the stack once. Done in 2 minutes.

By default all new resources in the account are tagged. Use VPC scoping if migration and non-migration workloads share the same account.
These services have no VPC association. Check to tag them all regardless of VPC scope (recommended for MAP eligibility).

Multiple Accounts (AWS Organizations)

Customer has multiple linked accounts under an AWS Organization. One StackSet command deploys to all accounts simultaneously.

Choose how the auto-tagger covers all accounts.

StackSet — Deploy to every account Recommended · Simplest

The same CloudFormation stack deploys to every specified account via a single StackSet. Each account gets its own Lambda that tags its own resources. No cross-account IAM. The template auto-discovers the organization structure — the customer never needs to know or enter any OU IDs.

1 command total. Enter specific account IDs to target only those accounts, or leave empty to cover every account in the org. The template figures out the rest automatically.

Central Lambda — Single account tags all others Advanced

One Lambda in a central account. Member accounts forward CloudTrail events to it via EventBridge. Lambda assumes a cross-account role to tag resources in each member account.

Requires per-account setup: Each member account needs an IAM role (MapAutoTaggerCrossAccount) and an EventBridge forwarding rule. Deploy these via StackSet or manually.
Note: AWS deploys SERVICE_MANAGED StackSets by Organizational Unit, not by account ID. The template auto-discovers your org's root OU and deploys to all accounts. The IDs you enter here are documentation only and appear in the deployment instructions.

Customer Prerequisites — the generated deploy.sh will verify these automatically before deploying:
  • CloudTrail must be enabled in the target region(s)
  • Deployer must have CloudFormation permissions (IAM, Lambda, EventBridge, SSM, SNS, SQS, CloudWatch)
  • Management account must have AWS Organizations enabled with CloudFormation StackSets trusted access
⚠️ Service Control Policy (SCP) Advisory
If the customer's organization uses SCPs, verify the following before deployment:
  • Tagging SCPs — ensure SCPs do not deny tag:TagResources, ec2:CreateTags, or other tagging actions for Lambda execution roles. The deploy.sh will run a post-deployment IAM simulation check, but SCPs require manual verification in the AWS Organizations console.
  • Mandatory tagging SCPs — if SCPs require tags to be present at resource creation time, this solution will not satisfy that requirement (tags are applied typically 60–90 seconds (up to 15 minutes during high-volume activity) after creation). The customer must exempt the tag key map-migrated from creation-time enforcement, or use a grace period.
