# 08 — Security

> ⚠️ Mirrored in `.kiro/steering/` and `.claude/rules/`. Run `npm run sync-rules` after edits.
> These are **hard constraints**. Treat violations as blocking.

## Data-safety hard rules (non-negotiable)

- **`delete.sh` must NEVER remove `map-migrated` tags.** Removing the solution preserves existing tags — MAP credits are permanent and irreversible. Deletion removes infrastructure (EventBridge, SQS, Lambda, IAM, SSM, alarms) only, never the tags on customer resources.
- **Preserve operational data on teardown** — log groups use explicit DeletionPolicy + retention (§1.74). Don't silently destroy data customers may need.

## Generated-script safety

The configurator generates shell scripts that customers run **with their own AWS credentials**. Therefore:
- **Single-quote containment** for any user-supplied value interpolated into shell (the `customerName` injection fix, U4/v20.5.2).
- **`lint_shell_injection.py` must pass** — it guards the generated scripts.
- **XSS hardening** on any HTML that renders user input (review-table XSS fix, v20.9.5).

## Least-privilege IAM

- IAM policies are generated by `.github/scripts/generate_iam.py` and gated by an IAM-completeness CI check.
- Grant only the tagging actions the covered services need — no broader wildcards.
- The customer owns all deployed roles (AWS Shared Responsibility Model). State this; don't assume the customer wants broad grants.

## Advisory scanners (Layer 1 CI)

`cfn-guard`, `cfn-nag`, and `bandit` run as advisory scanners (#21). Respect their findings.

## Secrets and boundaries

- **No secrets or credentials in the configurator** — it's a static client-side file.
- **Report security issues via the AWS vulnerability reporting page, never a public GitHub issue** (see `CONTRIBUTING.md`).
- **Respect SCP constraints** — the tagger cannot exceed org guardrails (documented in `LIMITATIONS.md`).
