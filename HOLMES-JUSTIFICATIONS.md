## Finding 1 — IAM: Resource `*` on Tagging Actions
**Scanners:** Checkov (CKV_AWS_108, CKV_AWS_111), cfn-guard, rubric
**Severity:** HIGH
**Affects:** All templates (single-account, multi-account accounts, org)

**Disposition: Accepted Risk — AWS platform constraint**

1. **`tag:TagResources` with `Resource: *`** — The AWS Resource Groups Tagging API does not support resource-level restrictions. AWS documentation explicitly states this. This is a platform constraint, not a design choice. Precedent: MAP Taggr (AWS-internal tool, AppSec-approved) uses the same pattern.

2. **150+ service-specific tagging actions with `Resource: *`** — The Lambda receives resource ARNs at runtime from CloudTrail events. It is architecturally impossible to pre-declare specific customer ARNs in an IAM policy for a general-purpose tagging solution. Each customer will create different resources with different ARNs.

3. **Condition statements absent** — IAM conditions scoped to specific VPCs or accounts would break multi-region and multi-account deployments. Scope filtering is enforced at the application layer: SSM config stores `scoped_account_ids` and `scoped_vpc_ids`, and the Lambda's `is_in_scope()` function enforces filtering at runtime.

4. **`MinimalReads` with `Resource: *`** — Read-only describe actions (`ec2:DescribeInstances`, `sts:GetCallerIdentity`, `ssm:GetParameters`) are required for the Lambda to look up resource details and read its own config. These cannot be scoped to specific resources because resource ARNs are unknown at deploy time.

Documented as accepted risk in THREAT-MODEL.md (Section 6).

## Finding 2 — IAM: Inline Policies
**Scanner:** cfn-guard (IAM_NO_INLINE_POLICY_CHECK)
**Severity:** LOW
**Affects:** AutoTaggerRole, BackfillRole, DeployRole

**Disposition: Accepted Risk — deliberate design**

Inline policies are used intentionally:
- This Lambda has a single, fixed purpose. There is no reuse case that would justify a managed policy.
- Inline policies are deleted when the IAM role is deleted (CloudFormation stack teardown), preventing orphaned permissions.
- The policy content is static and fully visible in the CloudFormation template — no hidden permissions.

AWS documentation acknowledges inline policies as appropriate for single-service, single-purpose roles.

## Finding 3 — Lambda Configuration
**Scanner:** cfn-guard (LAMBDA_INSIDE_VPC, LAMBDA_CONCURRENCY_CHECK, LAMBDA_DLQ_CHECK)
**Severity:** LOW
**Affects:** AutoTaggerFunction, BackfillFunction, DeployFunction

**Disposition: Accepted Risk — deliberate design**

**Lambda not in VPC (`LAMBDA_INSIDE_VPC`):** The Lambda exclusively calls AWS public APIs (CloudTrail, tagging, SSM, SQS). Placing it inside a VPC would require a NAT Gateway or VPC endpoints, adding ~$30–50/month per account with zero security benefit. The Lambda has no database connections or private network resources to protect. Documented in THREAT-MODEL.md.

**No reserved concurrency (`LAMBDA_CONCURRENCY_CHECK`):** Reserved concurrency is appropriate for Lambdas that must be throttled to protect downstream systems. This Lambda processes CloudTrail events and calls tagging APIs — neither downstream system requires throttling. Setting reserved concurrency would risk dropping tagging events during bursts, causing missed MAP credits.

**BackfillFunction has no DLQ (`LAMBDA_DLQ_CHECK`):** The BackfillFunction is a one-shot Lambda invoked once at deployment via a CloudFormation Custom Resource. It is not an event-driven function and has no ongoing event queue. DLQs apply to async Lambda invocations from event sources (SQS, SNS, EventBridge); they are not applicable to synchronous Custom Resource invocations. The main AutoTaggerFunction does have a DLQ (SQS RetryQueue → DeadLetterQueue).

## Finding 5 — Code Security: Input Validation
**Scanner:** rubric
**Severity:** MEDIUM
**Affects:** Lambda Python code in all templates

**Disposition: Accepted Risk — trust boundary is AWS CloudTrail**

The Lambda processes CloudTrail events delivered by AWS EventBridge — a fully AWS-authenticated, AWS-controlled pipeline. CloudTrail events cannot be forged by non-AWS principals (EventBridge verifies source via `SourceArn` condition on the Lambda permission). This is the trust boundary established in THREAT-MODEL.md.

Within this trust boundary, CloudTrail events have a well-defined and stable AWS-maintained schema. The Lambda uses `.get()` dict access with defaults throughout, preventing `KeyError` exceptions. The flagged patterns (`json.loads()` without try/except on specific fields) operate on data from the trusted CloudTrail pipeline.

For truly untrusted input (MPE ID, VPC IDs, account IDs entered in the configurator UI), validation is enforced:
- MPE ID: regex `^mig[a-zA-Z0-9]+$` at both JavaScript and CloudFormation `AllowedPattern` layers
- VPC IDs: validated against `vpc-[0-9a-f]{8,17}`
- Account IDs: validated as exactly 12 digits
- Dates: HTML `min`/`max` constraints

## Finding 7 — Data Security: SSM Type String
**Scanner:** rubric
**Severity:** MEDIUM
**Affects:** All templates

**Disposition: Accepted Risk — non-sensitive configuration**

The SSM parameter stores: MPE ID (a business identifier like `mig1234567890`), agreement start/end dates, scope mode, and account/VPC ID lists. None of these are credentials, secrets, API keys, or PII.

Using `SecureString` would require a KMS key to be pre-created before deployment (a prerequisite burden on the customer) and adds ~$1/month per account for KMS decrypt calls on every Lambda invocation — with no security benefit for this data classification. This decision is documented inline with a SECURITY NOTE comment in the template.

## Finding 8 — Data Security: Encryption and Key Management
**Scanner:** rubric
**Severity:** MEDIUM/HIGH
**Affects:** All templates

**Disposition: Accepted Risk — appropriate for sample code**

**SQS/SNS encryption:** Both queues and the SNS topic use AWS-managed keys (`alias/aws/sqs`, `alias/aws/sns`) providing encryption at rest. Customer-managed KMS keys require customers to pre-create keys, manage rotation, and grant cross-account permissions — inappropriate dependencies for sample code. AWS-managed keys satisfy encryption at rest requirements.

**CloudWatch Logs without KMS:** Lambda logs contain operational data only (resource ARNs, tag operation results, error messages) — no credentials, PII, or sensitive data. Customer-managed KMS for CloudWatch Logs requires pre-created keys and is not standard for operational logging in sample solutions.

**Key management documentation:** Key management strategy (rotation, access) for AWS-managed keys is handled by AWS and documented in AWS KMS documentation. Duplicating AWS service documentation in sample code is out of scope.

## Finding 9 — S3 Security: Staging Bucket
**Scanner:** rubric
**Severity:** MEDIUM
**Affects:** Multi-account deploy.sh

**Disposition: Accepted Risk — staging bucket for single-file artifact**

The S3 staging bucket (multi-account only) holds a single ~40KB CloudFormation template. Security controls applied:
- ✅ Block Public Access (all 4 settings)
- ✅ AES-256 encryption at rest
- ✅ HTTPS-only bucket policy (fixed in commit `4184376`)

**Access logging:** Access logging is designed for buckets with sensitive data access patterns requiring audit trails. A single-file deployment artifact bucket does not meet this threshold. Adding access logging requires a second destination bucket (another resource to manage).

**Versioning:** The bucket stores a generated CloudFormation template that is replaced on each redeployment. Versioning would accumulate old template versions indefinitely with no recovery use case. The source of truth is `configurator.html` in the repository.

These are documented as accepted risks in THREAT-MODEL.md (Section 6).

## Finding 14 — AWS Service Name Standards
**Scanner:** rubric
**Severity:** MEDIUM/HIGH
**Affects:** All files

**Disposition: Partial fix + false positive (CLI context)**

## Finding 15 — URL Validation
**Scanner:** rubric
**Severity:** MEDIUM
**Affects:** All files

**Disposition: False positive — runtime variables, not literal URLs**

The scanner flags these as invalid URLs:
- `https://sqs.{qr}.amazonaws.com/{qa}/{qn}` — Python f-string with runtime variables (`qr`, `qa`, `qn`)
- `https://bedrock-agentcore-control.{r}.amazonaws.com/tags/{encoded}` — Python f-string with runtime variables

These are not literal URLs — they are code template strings where `{variable}` and `${VARIABLE}` are replaced at runtime with real values. The scanner attempted to fetch them as static URLs, which correctly fails. These are valid patterns for their respective languages.

## Finding 17 — Multi-Account: DeployRole Permissions
**Scanner:** rubric
**Severity:** HIGH
**Affects:** `map-auto-tagger-org.yaml` only

**Disposition: Accepted Risk — architectural requirement**

The `DeployRole` in the org template grants `organizations:*`, `cloudformation:*`, and related permissions with `Resource: *`. This role is used exclusively by the Custom Resource Lambda that creates the CloudFormation StackSet.

Creating a SERVICE_MANAGED StackSet requires:
- `organizations:DescribeOrganization` / `ListRoots` / `ListAccountsForParent` — to discover the org root OU
- `cloudformation:CreateStackSet` / `CreateStackInstances` / `DescribeStackSet` — to create and manage the StackSet
- `iam:GetRole` / `PassRole` — CloudFormation StackSet service-linked role management

These permissions cannot be scoped to specific resources because the StackSet name and org root OU ID are not known at policy creation time (they are determined at deployment runtime). The Lambda runs once at deployment and the role is deleted with the stack.
